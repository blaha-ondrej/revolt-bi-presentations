Základy
Co je datový sklad
a proč ho stavíme?
DWH = jedno místo, kde se scházejí data ze všech systémů klienta — čistá, konzistentní a připravená k analýze.
Problém
Data jsou všude
ERP, CRM, e-shop, marketing, Excel soubory. Každý systém má svá data, svůj formát, svou logiku. Nikdo nemá celkový přehled.
Řešení
Jeden zdroj pravdy
DWH sbírá, čistí a modeluje data z všech zdrojů do jedné konzistentní vrstvy. Byznys pak pracuje s daty, na která se může spolehnout.
Výsledek
Rozhodnutí podložená daty
Reporty, dashboardy, ad-hoc analýzy, AI agenti — všichni čerpají ze stejného zdroje. Žádné nesrovnalosti, žádné dohadování.

L1 — Staging vrstva
Normalizace, čištění a standardizace. Data začínají dávat smysl.
ABC
Normalizace názvů sloupců
Vše přechází na snake_case. Konzistentní konvence pojmenování v celém DWH.
123
Správné datové typy
Datumy jako DATE/TIMESTAMP, čísla jako NUMBER, texty jako VARCHAR. Konec castování v každém dotazu.
CLN
Základní čištění dat
Trimování whitespace, unifikace NULL hodnot, deduplikace na úrovni zdroje.
MAP
Mapování kódů a zkratek
Stavové kódy ("A", "I", "P") se převedou na čitelné hodnoty ("active", "inactive", "pending").
-- L1: stg_orders.sql
SELECT
  order_id,
  customer_id,
  order_date::DATE          AS order_date,
  total_amt::NUMBER(18,2)  AS total_amount,
  CASE stat
    WHEN 'A' THEN 'active'
    WHEN 'C' THEN 'cancelled'
  END                       AS order_status
FROM {{ source('eshop', 'orders') }}

-- L2: fct_orders.sql
SELECT
  o.order_id,
  o.order_date,
  c.customer_segment,
  p.product_category,
  o.total_amount,
  o.total_amount * 0.3  AS gross_margin
FROM {{ ref('stg_eshop__orders') }} o
LEFT JOIN {{ ref('dim_customers') }} c
  ON o.customer_id = c.customer_id
LEFT JOIN {{ ref('dim_products') }} p
  ON o.product_id = p.product_id
Klíčový princip: Byznys logika patří do L2, ne do Power BI nebo Tableau. Pak ji máte na jednom místě, pod verzí, otestovanou.

Transformační nástroj
dbt — jak transformujeme data
dbt (data build tool) přináší do datového inženýrství to nejlepší z vývoje softwaru.
dbt ti dovoluje psát SQL SELECT dotazy a on za tebe vyřeší zbytek — vytvoří tabulky/views ve Snowflake, hlídá závislosti, testuje data, generuje dokumentaci.
SQL
Píšeš jen SQL
Každý dbt model = jeden SQL soubor s SELECT dotazem. dbt ho zkompiluje a spustí.
REF
ref() a source()

GIT

Definuješ očekávání nad daty — unikátnost, not-null, referenční integritu. dbt test je spustí a okamžitě upozorní na anomálie.
columns:
 - name: order_id
   tests:
     - unique
     - not_null
 - name: status
   tests:
     - accepted_values
CI
CI/CD Pipeline
Každý Pull Request automaticky spustí testy a build. Žádná změna se nedostane do produkce bez validace.
# GitHub Actions
on: pull_request
jobs:
  dbt-ci:
    - dbt build --select
      state:modified+
MAC
Makra & DRY kód
Jinja templating dovoluje psát znovupoužitelné SQL macra. Opakující se logika jednou — ne na 50 místech.
{{- macro cents_to_czk(col) -}}
  ({{ col }} / 100.0)::NUMBER(10,2)
{{- endmacro -}}

-- použití:
{{ cents_to_czk('amount') }} AS amount_czk
SEM
Sémantická vrstva
dbt MetricFlow definuje metriky jednou — revenue, conversion rate, CAC — a sdílí je konzistentně napříč BI nástroji a AI.
metrics:
 - name: revenue
   type: sum
   expr: total_amount
   model: fct_orders
   dimensions: [date, channel]
